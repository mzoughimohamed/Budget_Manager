# i18n (French + Arabic) Design Spec

## Goal

Add French and Arabic language support to the frontend. The user can switch language from the top nav (always visible) and from the Settings page. Arabic triggers a full RTL layout flip. The preference is saved in `localStorage`.

## Architecture

**Approach:** Custom `LanguageContext` — no new npm dependencies. A `t(key)` function backed by plain JS translation objects, language stored in `localStorage` key `'lang'`.

**RTL:** When Arabic is selected, `document.documentElement.dir = 'rtl'` and `document.documentElement.lang = 'ar'` are set. Tailwind's built-in `rtl:` modifier handles layout-specific flips (nav flex direction, stat cards, form rows). No Tailwind config changes needed — `rtl:` variant works out of the box against the `dir` attribute on `<html>`.

**No backend changes.** Language is a frontend-only concern.

## `LanguageContext` Interface

```js
// frontend/src/contexts/LanguageContext.jsx
const SUPPORTED = ['en', 'fr', 'ar']
const DEFAULT = localStorage.getItem('lang') ?? 'en'

export function LanguageProvider({ children }) { ... }

export function useTranslation() {
  const { lang, setLang, t } = useContext(LanguageContext)
  return { lang, setLang, t }
}
```

`t(key)` implementation:
```js
function t(key) {
  return translations[lang]?.[key] ?? translations['en'][key] ?? key
}
```

## `LangSwitcher` Component

```jsx
// frontend/src/components/layout/LangSwitcher.jsx
const OPTIONS = [
  { code: 'en', label: 'EN' },
  { code: 'fr', label: 'FR' },
  { code: 'ar', label: 'ع' },
]

export default function LangSwitcher() {
  const { lang, setLang } = useTranslation()
  return (
    <div className="flex gap-1">
      {OPTIONS.map(({ code, label }) => (
        <button
          key={code}
          onClick={() => setLang(code)}
          className={`px-2 py-1 rounded text-xs font-medium transition-colors ${
            lang === code
              ? 'bg-app-accent text-white'
              : 'text-gray-500 hover:bg-gray-100'
          }`}
        >
          {label}
        </button>
      ))}
    </div>
  )
}
```

## RTL Layout Changes

These are the specific Tailwind class additions needed for proper Arabic layout:

| Component | Class added |
|---|---|
| `TopNav` tabs container | `rtl:flex-row-reverse` |
| `StatsRow` grid (already CSS grid, auto-handled) | none needed |
| `TransactionTable` header row | `rtl:text-right` on left-aligned `th` cells |
| `TransactionForm` type toggle row | `rtl:flex-row-reverse` |
| `MonthCalendar` day-of-week header | `rtl:flex-row-reverse` (if using flex) |
| `IncomeSourcesTable` row flex | `rtl:flex-row-reverse` |
| `DayPanel` flex rows | `rtl:flex-row-reverse` |

Text alignment throughout: `dir="rtl"` on `<html>` makes the browser right-align text in all elements by default — no per-element changes needed.

## Edge Cases

**Missing translation key:** `t(key)` falls back to English, then to the raw key string. Nothing breaks if a key is missed.

**Language on first load:** Reads `localStorage` on context init. If no value, defaults to `'en'`.

**Number formatting:** Already uses `toLocaleString('fr-TN')` everywhere (TND currency). This stays unchanged — it's a currency display format, not a language setting.

**Category/income names:** User-entered data — not translated.

**Date formatting:** `date-fns` `format()` calls produce English month names (e.g. "Apr 21"). For French/Arabic, these would ideally use `date-fns/locale`. However, to keep scope contained, date strings remain in English/Latin numerals — this is acceptable for a personal finance app. Can be added later as an enhancement.
